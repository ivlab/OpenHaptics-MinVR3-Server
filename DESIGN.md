# Design Document: OpenHaptics-MinVR3-Server

## 1. Overall System Architecture

This document describes the architecture of the `OpenHaptics-MinVR3-Server`, a standalone application that provides network-based access to a force-feedback haptic device.

### 1.1. Core Components
The server is built on a client-server model and comprises several key components:

1.  **Main Application Loop (`main.cpp`):** The application entry point orchestrates the entire system. It uses GLUT to create a simple OpenGL 1.x graphics context for debugging and visualization. The main loop (`glutDisplay`) drives the entire process each frame: network I/O, haptic device polling, event processing, haptic rendering, and graphics rendering.

2.  **Network Layer (`MinVR3Net`):** A simple TCP-based networking layer built upon the MinVR3 library. It listens for a single client connection. All communication between the client and server is accomplished by serializing and sending `VREvent` objects.

3.  **Event Manager (`EventMgr`):** A central event bus that decouples event producers from consumers. Events are received from the network client or generated by the haptic device (e.g., button presses) and queued. The `EventMgr` processes this queue, dispatching events to registered listeners.

4.  **Haptic Device Abstraction (`Phantom` class):** This class encapsulates all direct interaction with the OpenHaptics library (both HDAPI and HLAPI). It is responsible for initializing the device, polling for stylus position/rotation and button state, and managing the haptic rendering frame (`hlBeginFrame`/`hlEndFrame`).

5.  **Modular Force Effects (`ForceEffect` base class):** The core of the haptic rendering logic is a collection of `ForceEffect` objects. Each effect is a self-contained, polymorphic class that implements a specific haptic sensation (e.g., `AmbientFriction`, `SurfaceConstraint`). These effects are registered with the `Phantom` class at startup. They are controlled (started, stopped, parameters modified) via `VREvent`s sent from the client. During the haptic rendering pass, the `Phantom` class iterates through its registered effects and calls their `DrawHaptics()` method.

### 1.2. Data Flow and Execution Model

The server's main loop follows a well-defined sequence:

1.  **Network Read:** Check for and accept new client connections. Read any incoming `VREvent` data from the connected client and queue the events in the `EventMgr`.
2.  **Device Polling:** The `Phantom` class queries the OpenHaptics driver for the latest stylus transform and button states. If the state has changed beyond a certain threshold, new `VREvent`s are generated and queued for the client.
3.  **Event Processing:** The `EventMgr` processes its queue, calling the appropriate callback methods on listener objects. This is how `ForceEffect` objects receive commands from the client to start, stop, or change parameters.
4.  **Haptic Rendering:** The `Phantom` class begins the haptic frame. It sets up the haptic coordinate spaces and then calls `DrawHaptics()` on each active `ForceEffect`. Inside this method, the effects make OpenHaptics API calls to generate forces.
5.  **Graphics Rendering:** A simple graphics representation of the haptic scene is drawn for debugging. This includes a wireframe of the device workspace, a cursor for the stylus, and any debug visualizations provided by the `ForceEffect`s.
6.  **Network Write:** Any events generated during the frame for the client (e.g., stylus position updates) are serialized and sent over the network.

## 2. Coordinate System Architecture

To keep the server's implementation as simple and predictable as possible, the responsibility for all coordinate system conversions lies with the client application. The server operates exclusively in a single, raw coordinate system.

### 2.1. Server Coordinate System: "Touch Space"
The server and the OpenHaptics library operate in a **right-handed, Y-Up** coordinate system where the units are **millimeters (mm)**. This is referred to as **Touch Space**. This space is the raw, off-center coordinate system reported by the OpenHaptics driver. The dimensions of this space are defined as constants in `phantom.cpp` and can be queried by the client to facilitate transformations.

### 2.2. Client Responsibilities

The client application is responsible for all transformations required to map its data into the server's raw Touch Space. The server's coordinate system is intentionally simple to make this contract clear.

The following describes the implementation strategy used by the example Unity client. Future clients built with other toolkits (e.g., native C++, Unreal Engine) must perform equivalent transformations to match the server's coordinate system (right-handed, Y-up, millimeters, off-center).

#### Example: Unity Client Implementation

1.  **The `HapticWorkspace` Anchor:** A `HapticWorkspace` GameObject acts as the anchor for the haptic device in the Unity scene. Its `transform` (position, rotation, and scale) defines the complete mapping from the server's raw Touch Space to Unity's World Space.

2.  **Unit Conversion via Scale:** By setting the `HapticWorkspace`'s scale to `(0.001, 0.001, 0.001)`, the conversion between Unity's meters and the server's millimeters is handled implicitly by Unity's transform hierarchy.

3.  **Data to Server:** To send a point to the server, the Unity client takes a **Unity World Space** coordinate, transforms it into the **local space** of the `HapticWorkspace` (which implicitly converts it to millimeters), and then flips the Z-axis to convert from left-handed to right-handed coordinates.

4.  **Data from Server:** The server sends stylus position and rotation data back to the client in its raw Touch Space. The Unity client performs the inverse transformation: it flips the Z-axis of the incoming data (right-to-left handed) and then transforms this point from the `HapticWorkspace`'s local space into Unity World Space.

### 2.3. The `ModelToWorld` Transform (Deprecated)

The server no longer accepts a `ModelToWorld` transform from the client. All transformations are now handled exclusively on the client side, anchored by the `HapticWorkspace` GameObject. This provides a clearer, more robust, and more intuitive model for developers using Unity.

## 3. Guidelines for Implementing New Force Effects

The server is designed to be extensible through the `ForceEffect` polymorphic base class. To maintain consistency and stability, new effects should adhere to the following guidelines.

### 3.1. Create a New `ForceEffect` Subclass
1.  Create a new header and source file (e.g., `my_effect.h`, `my_effect.cpp`).
2.  The class must inherit from `ForceEffect`.
3.  The constructor should call the base class constructor, passing its `name` and the `event_mgr`. It should then register for any `VREvent`s needed to control the effect (e.g., start, stop, update parameters).
4.  Implement the `DrawHaptics()` and `DrawGraphics()` virtual methods.

### 3.2. Implement the `DrawHaptics()` Method
This method is the core of the haptic rendering logic. It is called once per haptic frame from within `Phantom::DrawHaptics()`, which is already wrapped by `hlBeginFrame()` and `hlEndFrame()` in `phantom.cpp`.

*   **For Geometry-Based Effects:** If the effect can be described by standard shapes (points, lines, triangles), use the HLAPI's shape-drawing commands (e.g., `hlBeginShape()`, `hlEndShape()`). This is the simplest approach and is used by `PointConstraint`, `LineConstraint`, and `SurfaceConstraint`.
*   **For Custom Mathematical Forces:** If the effect requires a custom force calculation (like the tangent plane approximation), it must not use `hlBeginShape()`/`hlEndShape()`. Instead, it should use an HLAPI custom effect callback.
    1.  In the constructor, generate an effect ID with `hlGenEffects(1, &effect_id_)`.
    2.  Define a static `HLCALLBACK` function for the force calculation. This function will receive the device state from the servo loop.
    3.  In response to a "start" event from the client, call `hlEffectd(HL_EFFECT_CALLBACK, &MyCallback)` and other `hlEffect*` functions to set parameters, then start the effect with `hlStartEffect(effect_id_)`.
    4.  The `DrawHaptics()` method for this type of effect may be empty, as the callback handles the force generation.

### 3.3. Register the New Effect
In `main.cpp`, include the new effect's header. Then, inside the `registerForceEffects()` function, create an instance of your new effect and pass it to `phantom->RegisterForceEffect()`.

## 4. Example Force Effect: Remote Surface Haptic Rendering via Local Tangent Plane Approximation

The modular `ForceEffect` architecture allows for new haptic sensations to be added to the server. The following section details the design for a new, advanced effect for rendering complex remote surfaces that are not defined by local geometry.

### 4.1. Executive Summary & Problem Statement
This effect enables high-fidelity haptic rendering of complex 3D surfaces where the primary geometric scene data and collision queries reside on a remote **Application Machine**, while the force-feedback loops run on the local **Haptic Server**.

#### The 1000Hz Constraints
1.  **Haptic Update Boundary:** The physical haptic device driver requires a strict **1000Hz (1ms)** update rate. Any drops below this frequency cause severe mechanical instabilities, device buzzing, or emergency safety cutoffs.
2.  **Network Bandwidth Boundary:** Network update frames from the client typically arrive asynchronously at **30Hz to 60Hz** with varied packet jitters and network latencies (10–50ms).
3.  **State Mutation Cost:** Dynamically modifying, deleting, or rebuilding high-level geometric shapes (e.g., calling OpenHaptics `hlBeginShape`/`hlEndShape`) inside the 1000Hz loop introduces severe context execution stalls and is not feasible for geometry streamed over the network.

#### Solution: Hybrid Local Tangent Plane Approximation
Rather than streaming raw vertex data to rebuild a local mesh, this `ForceEffect` relies on an analytical abstraction. The remote Application Machine performs its own collision detection against its local scene. It then streams a simplified, stateless representation of the surface at the point of contact: an infinite **local tangent plane** defined by a single 3D **Contact Point** and an associated 3D **Surface Normal Vector**. The Haptic Server receives these parameters at a low frequency and uses them inside a continuous, high-frequency custom force calculation.

### 4.2. System Architecture & Topology

The data pipeline is split across the client, network, and server:

*   **Application Machine (Client, 30-60Hz):**
    *   Performs collision detection (e.g., raycasts) against its local scene.
    *   Resolves the exact contact point and surface normal.
*   **Network Layer:**
    *   Encapsulates the point and normal into a packet and sends it to the server.
*   **Haptic Server (1000Hz):**
    *   A network thread receives the packet.
    *   The main haptic loop uses the point and normal data to compute a custom spring-damper force.

#### 4.2.1. Transport Layer Specification
* **Protocol:** UDP (User Datagram Protocol). In order to reduce network latency overhead, TCP retransmissions are strictly forbidden; stale lost frames must always be dropped in favor of the newest incoming spatial tracking payload.
* **Network Thread Isolation:** Inbound datagram reception must run asynchronously from the main haptic processing thread inside an independent background context, protecting state boundaries via high-efficiency synchronization primitives.

### 4.3. Mathematical Foundations & Constraints

#### 4.3.1. Force Formulation (Spring-Damper Model)
When the physical device's stylus penetrates behind the boundary layer of the analytical infinite plane, the haptic proxy generates counter-acting feedback forces calculated via a classic spring-damper formulation:

`Force = Normal * max(0, (-Stiffness * Depth) - (Damping * NormalVelocity))`

Where:
* $d$: Penetration depth, determined by projecting the relative stylus displacement onto the plane's unit normal vector: $(\vec{P}_{stylus} - \vec{P}_{contact}) \cdot \vec{N}$.
* $v_n$: Device velocity component aligned with the tracking path normal: $\vec{V}_{stylus} \cdot \vec{N}$.
* $k$: Localized spring stiffness parameter.
* $c$: Surface damping value used to minimize structural oscillations.

#### 4.3.2. Vector Blending via Spherical Linear Interpolation (Slerp)
Linear interpolation (Lerp) on a normal vector transitioning across angles diminishes the structural length of that vector (e.g., dropping vector magnitude down to $\approx 0.707$ at a halfway point of a 90-degree intersection). Because force calculation maps directly to the normal length, a standard Lerp will inject an unwanted **30% drop in perceived surface stiffness** as users trace over mesh curvature.

To ensure strict material consistency, normal directional changes must be blended over time via **Spherical Linear Interpolation (Slerp)**, forcing the directional normal vector to hold a rigid unit scale of $1.0$ across all rotational points.

### 4.4. Implementation Strategy

This effect will be implemented as a new `TangentPlaneConstraint` class inheriting from `ForceEffect`. It will follow the "Custom Mathematical Force" pattern described in the guidelines above.

#### 4.4.1. Class Structure
*   **Member Variables:** The class will store an `HLuint effect_id_` for the custom effect. It will also hold thread-safe member variables (e.g., using `std::atomic` or mutexes) for the current `contact_point` and `surface_normal` received from the network.
*   **Constructor:** The constructor will call `hlGenEffects(1, &effect_id_)` to reserve an ID for the custom effect. It will register listeners with the `EventMgr` for events that update the contact point and normal.
*   **Event Handlers:** Methods bound to network events will update the internal contact point and normal variables in a thread-safe manner.
*   **`DrawHaptics()`:** This method will be empty. The haptic rendering is offloaded to the callback.
*   **`DrawGraphics()`:** This method will render a visual debug representation of the current tangent plane (e.g., a quad at the contact point oriented by the normal).

#### 4.4.2. Custom Force Callback
A static `HLCALLBACK` function will be defined to perform the force calculation.

1.  **Signature:** `void HLCALLBACK compute_tangent_force(const HDdouble force[3], const HDdouble position[3], const HDdouble velocity[3], HDdouble xform[16], HDdouble result[3], void *userdata)`
2.  **`userdata`:** The `this` pointer of the `TangentPlaneConstraint` instance will be passed as `userdata`.
3.  **Logic:** Inside the callback, the function will:
    *   Safely read the latest `contact_point` and `surface_normal` from the `TangentPlaneConstraint` instance.
    *   Use the `position` and `velocity` arguments (provided by the servo loop) to calculate the penetration depth `d` and normal velocity `v_n`.
    *   Compute the final force vector using the spring-damper model.
    *   Write the 3D force vector into the `result` array.

This approach correctly injects a custom mathematical force into the OpenHaptics servo loop at 1000Hz without interfering with other geometry-based effects and while safely handling state updates from the lower-frequency network thread.

### 4.5. Critical Engineering Edge Cases

#### 4.5.1. Mechanical "Snapping" and Jolts
When tracking quickly over sharp transitions or adjacent triangles with highly disparate angles, incoming normals can jump instantly by up to 90 degrees. This creates massive accelerations in immediate force metrics ($\Delta F/\Delta t$), triggering mechanical system fault shutdowns.
* **Strategy:** Introduce an explicit alpha-blending factor inside the Slerp and position tracking matrices to smoothly filter raw data changes over a rolling timeline.

#### 4.5.2. Pop-Through and Tunneling Artifacts
If an operator applies high force vector speeds, the stylus coordinate could punch completely past the boundaries of the plane depth definition before the network layer or the application machine finishes a fresh raycast intersection query.
* **Strategy:** Enforce a strict **Distance Falloff Threshold**. If the spatial distance delta between the current haptic device coordinate and the targeted network coordinate breaks past a designated safety window ($>15\text{mm}$), the server must execute automated force shading (smoothly ramping feedback down to zero) to avoid violent physical corrections or rapid unexpected snaps.
